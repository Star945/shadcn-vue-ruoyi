# RuoYi Modern 设计需求文档

## 1. 文档目的

本文档用于统一当前项目的产品目标、设计边界、技术约束和后续维护要求，作为前后端持续开发、联调验收和开源协作的共同基线。

本文档不替代接口文档和代码实现，重点回答 4 个问题：

1. 这个项目要解决什么问题
2. 当前版本已经覆盖到什么程度
3. 后续开发时哪些设计原则不能被破坏
4. 维护者应该如何在现有架构上继续扩展

## 2. 项目背景

原始需求是以若依后台的业务结构和接口语义为基础，使用 `shadcn-vue + Tailwind CSS` 重构前端，并提供一套可对接、可替代 Java 若依后端的 NestJS 配套服务。

项目最终形态不是“逐像素复刻原版若依”，而是：

- 保留若依的后台工作流、菜单结构、权限模型和接口契约
- 用更现代的组件和主题体系重做前端体验
- 用 NestJS + Prisma 提供可落地的配套后台
- 在数据库可用与不可用两种情况下都能支持前后端联调

## 3. 目标与范围

### 3.1 总体目标

- 提供一套可运行的现代化若依后台前端
- 提供一套与前端契约兼容的 NestJS 后端
- 支持真实接口联调、开源展示和后续二次开发
- 降低共享组件、主题和权限逻辑的维护成本

### 3.2 本期范围

#### 前端范围

- 登录、注册、锁屏、401、404
- 动态菜单、面包屑、标签页、路由守卫
- 系统管理、系统监控、系统工具主要页面
- 主题系统、布局设置、移动端适配
- 表格、查询区、弹窗、操作栏、日期组件、富文本、头像裁剪等共享能力

#### 后端范围

- `login / register / logout / getInfo / getRouters / captchaImage / unlockscreen`
- `system/user / role / menu / dept / post / dict / config / notice`
- `monitor/online / logininfor / operlog / job / job-log / cache / server / druid`
- `tool/gen`
- `common/upload`
- CSV 导出
- Swagger 文档

### 3.3 明确保留的差异

以下部分允许与原版若依存在差异：

- 视觉主题和布局风格
- 表单构建页的产品形态
- Druid 的 Nest 兼容实现
- Swagger 的嵌入方式

## 4. 角色与使用场景

### 4.1 主要角色

- 平台管理员：维护用户、角色、菜单、部门和系统配置
- 运维人员：查看日志、在线会话、任务、缓存和服务状态
- 业务管理员：维护公告、字典、参数等后台数据
- 开发者：通过代码生成、接口文档和表单构建辅助开发

### 4.2 关键场景

- 管理员登录后台并进入对应菜单
- 基于权限控制访问页面和按钮
- 完成标准 CRUD、导出、分配和状态切换
- 在监控中心查看系统状态并执行必要管理操作
- 在工具模块上传文件、生成代码、查看接口文档

## 5. 功能需求

### 5.1 认证与会话

- 支持登录、注册、退出登录、锁屏解锁
- 支持获取用户信息和动态路由树
- 支持登录日志记录和在线会话管理
- 支持登录状态失效后的统一处理

### 5.2 菜单与路由

- 前端导航以 `/getRouters` 为主数据源
- 支持后端返回的树形菜单直接驱动侧栏和顶部导航
- 支持隐藏页、旧路径、组件字符串变体的兼容映射
- 支持路由级权限守卫和标签页管理

### 5.3 系统管理

- 用户、角色、菜单、部门、岗位、字典、参数、公告具备完整查询和 CRUD 能力
- 支持角色授权、用户分配角色、角色分配用户
- 支持树表、树筛选、状态切换、导出等典型若依后台行为

### 5.4 系统监控

- 在线用户管理与强退
- 登录日志、操作日志查询与导出
- 定时任务、调度日志管理
- 缓存监控、缓存列表、服务监控
- Druid 兼容页访问

### 5.5 系统工具

- 代码生成列表、导入表、预览、下载、同步
- Swagger 文档展示
- 表单构建工作台
- 文件上传

### 5.6 主题与布局

- 支持明暗模式
- 支持主题色和圆角半径调整
- 支持 TopNav、TagsView、固定 Header、Logo、动态标题、底部版权开关
- 支持本地持久化

## 6. 非功能需求

### 6.1 可维护性

- 页面逻辑尽量下沉到共享组件和共享工具，而不是复制页面模板
- 路由兼容、权限兼容、滚动策略等统一收敛到单点文件维护
- 新增页面必须优先复用现有共享组件
- 共享组件应通过配置和插槽扩展，而不是频繁分叉复制

### 6.2 可扩展性

- 后端模块按若依语义拆分，保证后续逐模块真实化
- 前端保留少量本地独立路由，其余尽量由后端菜单树驱动
- 主题系统和共享组件要允许继续扩展，不依赖硬编码视觉值

### 6.3 联调稳定性

- 数据库不可用时，后端可自动回退 mock，保证前后端仍可联调
- 上传、导出、Swagger、Druid 需在开发环境代理下正常访问
- 用户可见的错误信息必须是可读中文，避免返回乱码

### 6.4 移动端可用性

- 查询区、表格、树筛选、弹窗和顶部导航必须具备小屏下的可操作性
- 滚动区使用统一滚动策略，避免局部退回原生粗滚动条

## 7. 架构要求

### 7.1 前端架构要求

- 共享业务能力集中在 `src/components/admin`
- 共享应用壳层能力集中在 `src/components/app`
- 路由兼容与后端映射集中在 `src/router`
- 导航状态和菜单树集中在 `src/stores/navigation.ts`
- 请求与鉴权逻辑集中在 `src/utils/request.ts`

### 7.2 后端架构要求

- 模块边界保持与若依主要业务域一致
- 优先使用 Prisma 实现真实数据访问
- `safeCall` 机制用于数据库优先、mock 回退
- 认证、上传、导出、日志拦截等公共能力集中在 `common` 与 `auth`

## 8. 设计约束

- 不得重新引入“所有页面配置集中到单文件统一渲染”的旧模式
- 不得在页面中重复实现操作栏、查询区、弹窗壳、日期选择等共享样式
- 不得让菜单分组再次回到前端固定写死的方式
- 不得新增依赖页面私有约定的大量硬编码颜色和尺寸
- 不得把高频维护逻辑分散到多个文件重复实现

## 9. 代码注释与维护要求

### 9.1 注释策略

只在以下场景添加注释：

- 共享组件中存在不直观的状态推导
- 为兼容若依协议做了非显而易见的映射
- 为了联调稳定性引入数据库回退、滚动策略、权限策略时
- 某段逻辑如果删掉或改写，容易引起行为回归

禁止添加“变量赋值型”注释，例如“给变量赋值”“调用接口获取数据”。

### 9.2 维护重点文件

后续维护优先关注这些文件：

- `src/components/admin/AdminDataTable.vue`
- `src/stores/navigation.ts`
- `src/router/backend-route-map.ts`
- `src/components/admin/AdminDialogContent.vue`
- `apps/server/src/auth/auth.service.ts`
- `apps/server/src/common/interceptors/operlog.interceptor.ts`
- `apps/server/src/tool/gen/gen.service.ts`

### 9.3 新需求落地原则

- 先找共享能力能否承接
- 再决定是否新增扩展点
- 最后才考虑页面私有实现

## 10. 验收基线

### 10.1 前端验收

- 登录、注册、锁屏、错误页可正常访问
- 菜单、面包屑、标签页与后端路由树一致
- 系统管理核心页面可完成主要 CRUD
- 监控和工具页面可正常访问并展示真实接口结果
- 主题、布局、移动端、滚动体验达到可用状态

### 10.2 后端验收

- `apps/server` 可独立构建启动
- `getInfo / getRouters` 正常工作
- 系统管理、监控、工具模块主链路可联调
- 上传和导出返回真实文件结果
- Swagger 与 Druid 兼容页可访问

## 11. 后续迭代建议

### 11.1 第一优先级

- 做一轮浏览器级逐页联调验收
- 补数据库迁移和初始化说明
- 增加部署文档和环境说明

### 11.2 第二优先级

- 为高频共享组件补单元测试或最小回归脚本
- 继续收少量边角页面与行为差异
- 对工具模块做更深的产品化收口

## 12. 结论

当前项目已经完成主开发阶段，后续工作的重点不再是大规模重构，而是围绕本文档约束进行联调、交付和长期维护。
